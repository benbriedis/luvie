
Design decisions
================
This document outlines reasons for some of the decisions I've made, and is intended to 
help inform future developers to maintain functionality and consistency in the application.


1. Re the use of "initial dots" for parameter rubberbands in both the song editor and pattern editors...
   could be removed, but I think its better to have them for when we loop segments 
   and otherwise reposition the playhead. In these cases it is fairly clear how the parameters are to be handled.

1. Re use of trailing rubber band after the final parameter dot... Could be removed, but its clearer that the value has
   been changed and we can see what it is, so the extra clutter is likely worth it.

2. Re clashing of parameter values:
	Sometimes an instrument may receive multiple contradictory parameter changes at the same moment. In these cases changes 
	from the Song Editor parameter lanes receive the lowest priority, and for the patterns the priority increases with the 
	pattern track number as read top to bottom on the UI (eg a change in track 7 trumps a change in track 2).       
	My thinking: patterns are more specific than the song and should probably take priority for that reason.
	My thinking: between patterns its good to have a simple rule. Also the lower patterns were more likely added later so the 
	             user is more likely to expect them to take priority. 

FOLLOW
======
1. At the moment I'm deliberately omitting follow for patterns as it think it would be too jumpy.
2. In the song editor I'm scolling page by page rather than keep the playhead in the center, 
   which should give a stabler experience I think, and be lighter on the CPU.

TRANSPORT SETTINGS
==================

The user will almost always want to use the same transport, so save the selection to the project.
   DONT fallback to "Internal" if Jack is not present.
   Revert to polling when Jack is selected but the server is not present.
   Don't hide Jack as a possible transport if the server is not up... that is likely to make starting up fussy
      and lead to bad settings being saved in projects. The polling loop approach seems to be the best approach.
   New projects need a "Setup" window to popup, otherwise we're just guessing what the user wants.

 I've disallowed the use of --test with a session to prevent the accidental changing of settings in
 existing projects. Could support it in future if useful.

BACKUPS AND VERSIONING
======================
- Generally Ardour and NSM should look after these.

- In standalone mode it reasonable I think for the external programs and the user to 
  look after most of this. 'Export' and 'Import' is enough. Some people use Git.

- Nice to have though: some basic backups.
  - Can store versions INSIDE the project file for convenience.
    MAYBE just in standalone? If in NSM or Ardour code cause confusion.
  - Could save the last few exit states (not saves)
	  + A copy at least an hour old
	  + A copy at least a day old
	  + Maybe a copy at least 3 days old
  - Then need to be able to swap to using the old version and saving the current version.

ALSA and Jack connections?
==========================
Is it ever really worth mixing ALSA and Jack connections? Maybe, maybe not...
  Nonetheless from a S/W design perspective it would seem to be cleanest and least
  presumptive to allow them to be mixed, 
I have added a default connection type and I specify at this startup (again, not presumptive)
  and I'm pretty happy with this solution.


Alternate plugin architectures
===============================
1. Just use the Ardour plugin (LV2 or Clap) primarily or entirely for saving the session.
   Have Jack do the heavy lifting and communicate back using Jack.
   Has the advantage of being more like the standalone application - which would be more independent
       and possibly less work to maintain.
   Would it sit in a separate process?
   Would ports look the same?
   A negative: requires Jack if we are to use as a plugin. One of the advantages of using a plugin 
               is that it reduces external requirements like jack.
	- Could try to extend LV2 to support transport commands.
   
2. Clap apparently supports transport commands, so may be able to remove Jack dependencies when used within a plugin.
   DONT THINK Ardour supports Clap at the moment.

MISSION STATEMENT
-----------------
Luvie has aspects of being an instrument and aspects of being a piece of music notation s/w.
Don't want to get too carried away with the second aspect - could have endless time signatures
etc, but its more important to be musical about things. 
(Do need to be able drive Jack I guess, inc the time master, and maybe fit in with other 
requirements so there may be a bit of compromise required.).


CONVERTING BETWEEN CHORDS OR BETWEEN SCALES
===========================================
The rules used as: 
1. Keep the root note the same.
2. Match notes up from there  cf C ==> Cdim triad
     1. c => c
	 2. e => e flat
	 3. g => g flat
3. Many chords extend over 2 (strictly speaking they can go further, but I'm not bothering with those).	 
   In the main pattern editor chords are allowed to ascend up to 2 octaves within a chord/scale block. 
   If they do enter a second octave then the note pitches will drop back to a lower pitch when going up
   a chord/scale octave.

So the matching algorithm is very simple (an advantage) and isomorphic across conversions (another advantage).
In practice it works quite well because with chords:
	- Almost all have a 3rd or 3rd equivalent (eg 2 or 4).  
		Power chords are an exception, but they're not very interesting for a program like this.
	- Almost all have a 5th or 5 equivalent.
	- Almost all chords with more than 3 notes have a 7th or equivalent.
	- Pretty much everything else is a bonus.
Similarly the most common scales have 7 notes with equivalent notes.
Also its reasonable to support conversions between like scales (eg pentantics) or chords,
   but matching up very different ones is probably not a reasonable thing to expect.
