The full set of realistic options:

  1. JACK transport.  Bidirectional, clean. The strong choice for serious Linux DAW users.

  2. MIDI Machine Control (MMC)

  A standard SysEx protocol that many DAWs (Ardour, Reaper, Bitwig, Cubase…) implement for
  transport control. Commands include Stop, Play, and crucially Locate — which repositions the
  transport to a specific SMPTE timecode.

  Luvie already has a MIDI output port, so the DSP could send MMC SysEx events on it with no
  extra LV2 ports. Play/stop is trivial. Locate requires converting bars/beats → seconds →
  SMPTE frames, but Luvie already knows BPM and sample rate from time:Position, so the maths
  are there.

  The friction: the user has to manually route Luvie's MIDI out back to the host's MMC input
  (in Ardour: Preferences → MIDI → MMC, then patch in the JACK/ALSA MIDI graph). This is doable
   but awkward, and some users won't know to do it.

   BENS THOUGHT: may be useful in cases where Jack is not available, and/or we not using LV2.

  3. OSC (Open Sound Control)

  Ardour has a full OSC interface including /ardour/transport_play, /ardour/transport_stop,
  /ardour/locate. But this requires UDP networking even on localhost, needs to know Ardour's
  OSC port, is entirely Ardour-specific, and has no standard LV2 integration point. Not worth
  pursuing.


❯ So the main advantage of using here Jack seems to be that Luvie can auto-detect it's       
presence and connect to it. Is there any way Luvie could perform the patching required for   
the MMC solution?                                                                            
                                                                                             
● Technically yes — if JACK is present, the JACK API has jack_get_ports() to enumerate MIDI  
  ports and jack_connect() to wire them up. Luvie could search for a port matching a pattern 
  like "ardour:MMC*" and connect automatically.                                              
   
  But this reveals the circular problem: auto-patching MIDI via JACK requires JACK, at which 
  point you'd just use JACK transport directly and skip MMC entirely.
                                                                                             
  In non-JACK environments (PipeWire native mode, plain ALSA) the ALSA sequencer API can     
  enumerate and connect ports, but it's significantly more complex, less reliable, and you'd
  still need to guess the right port name pattern per-host. It also only helps hosts that    
  expose a named MMC input port in a discoverable way — not all do.

  So the realistic picture collapses to:

  - JACK/PipeWire-with-JACK-compat (covers most serious Linux DAW setups): use JACK          
  transport, auto-detect, auto-connect. MMC irrelevant.
  - Everything else: no reliable auto-patching, MMC requires manual user setup, and the user 
  base is probably small enough that "configure it yourself" is acceptable documentation.    
   
  The main practical advantage of JACK transport isn't really the auto-detection — it's that 
  it's a transport protocol rather than a MIDI protocol routed through the DAW's instrument 
  graph. JACK transport operates at the session level; both Ardour and Luvie are peers       
  talking to the same JACK server. MMC requires Ardour to be specifically configured to act
  on it, which is an extra step regardless of patching.

  Given all this, JACK transport as primary with a manual-setup MMC fallback documented for  
  non-JACK users seems like the right call.

● That's a sensible design. A few things worth keeping in mind for when you implement it:

  JACK client in the UI, not the DSP

  The UI process is the right place to open the JACK client (just for transport — not audio). The
  DSP already runs inside the host's JACK graph and you don't control that client. Opening a
  separate lightweight JACK client from the UI just for jack_transport_* calls is the standard
  approach.

  PipeWire compatibility

  On modern Ubuntu/Fedora/etc., libjack is typically provided by pipewire-jack and
  jack_client_open() will succeed even without a "real" JACK server running. So your auto-detect
  will work on most current Linux desktops, not just dedicated JACK setups. Worth testing, but it's
   likely better coverage than you might expect.

  Ardour needs to be a JACK transport slave

  Ardour defaults to being transport master. The user needs Session → Properties (or Preferences,
  depending on version) → Sync → "JACK Transport" to make it follow external transport commands.
  Worth documenting as a one-time setup step.

  Seek granularity

  jack_transport_locate() takes a frame position (samples). You'll need the sample rate and current
   BPM/time-sig to convert a bar/beat position to frames. The DSP already receives all of this via
  time:Position, so you'll need to forward it to the UI (via the notify_out port discussed earlier)
   before seeking can work correctly. Play/stop needs nothing extra.



Missing first note problem 
==========================
When using Ardour + Jack Ardour appears to add a latency or synchronization correction to the plugin.
When we rewind the transport we end up with a frame >  0. If there is note there it won't play.

Ideally I guess we'd have Luvie make its own latency adjustment, adding an invisible 5136 or so frames
on to the start. If we do this how would we calculate it? Options: 
1. Just add a blank bar to the start of a song.
   Could also have a 'song start' marker and rewind back to that.
2. When we get a reposition event with bar=0 use the beat value to calculate the frame offset.
   A bit problematic, but likely to work 99.9% of the time and would be of minimum fuss to implement.
   We'd need to the DSP to pass the info on the the UI I guess.
3. We could give the DSP its own Jack connection and obtain the information that way.
   Probably using jack_port_get_latency_range(port, JackPlaybackLatency, …) or jack_transport_query().
   WATCH OUT: I still want to use LV2 to actually provide the current transport position.
   A bit annoying as I feel its the LV2 host's job to give us this info, or not cause it in the first place.
   Still this is probably the most correct solution, if a bit heavyweight.
4. One could rewind and measure the latency. Possibly a bit too Ardour-specific a solution.
